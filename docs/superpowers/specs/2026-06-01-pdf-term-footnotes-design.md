# Термины-попапы как сноски в PDF

Дата: 2026-06-01
Репозиторий: `pdf-generator`

## Проблема

Термины (попапы) в вебе раскрываются по клику на выделенное слово. В PDF клик ничего
не делает — содержимое попапа недоступно читателю. Нужно перенести содержимое
определений в текст PDF так, чтобы оно не терялось и было удобно навигировать.

Пример: попап про «255 уровней» работает на сайте, но не в PDF.

## Цель

1. Все определения терминов выводятся блоком **в конце каждой страницы** (раздела).
2. Клик по термину в тексте ведёт **к его определению** (переход вперёд).
3. Блок определений **прижат к низу** PDF-страницы, чтобы при коротком контенте
   определения не висели сразу под текстом с пустым пространством снизу.

Вне рамок (осознанно отброшено для простоты):
- Объединение повторов: один и тот же термин, встреченный несколько раз, даёт
  **несколько** сносок (по одной на вхождение). Дедупликации нет.
- Обратная навигация (сноска → термин) **не делается**.
- Сноски внизу каждого **физического** печатного листа (требует смены движка
  рендеринга на Prince/WeasyPrint) — не делаем.

## Контекст кода

Итоговый PDF — один HTML, склеенный из блоков статей:

```html
<div class="pdf-page-wrapper" data-page-break="true">…контент статьи…</div>
```

(см. `packages/cli/.../pdf-page/utils.ts → joinPdfPageResults`). У каждого
`page-break-after: always`, то есть каждая статья начинается с нового листа.
Эти блоки — прямые соседи, склеены через `.join('')`.

Разметка терминов (плагин `@diplodoc/transform`, `setTermAttrs`):

- Термин в тексте:
  `<i class="yfm yfm-term_title" term-key=":KEY" role="button" aria-controls=":KEY_element" tabindex="0" id="…">ТЕКСТ</i>`
- Определение:
  `<dfn class="yfm yfm-term_dfn" id=":KEY_element" role="dialog">…CONTENT…</dfn>`

Связь термина и определения: `aria-controls` термина == `id` определения == `:KEY_element`.
Фолбэк, если у `<i>` нет `aria-controls` (напр. в код-блоках): `term-key` + `_element`.

Обработка терминов делается в `pdf-generator/src/generate/utils.ts → termsProcessing(html)`,
которая вызывается из `generatePdfStaticMarkup` над контентом `<main>`. Подход —
строковая обработка (регэкспы), без HTML-парсера и новых зависимостей.

## Решение

### Алгоритм `termsProcessing(html)`

**Шаг 0. Карта определений и удаление инлайновых `<dfn>`.**
Одним проходом собрать `defsMap[":KEY_element"] = CONTENT` из всех
`<dfn …class="…yfm-term_dfn…"… id="…">CONTENT</dfn>` (нежадно до первого `</dfn>`).
Затем удалить эти `<dfn>` из html — они переедут в блок сносок.

**Шаг 1. Разбить на страницы.**
Разбить html по разделителю `<div class="pdf-page-wrapper"`. Каждый блок
обрабатывается независимо. Фолбэк: если ни одного `.pdf-page-wrapper` нет —
весь html обрабатывается как одна страница.

**Шаг 2. Глобальный счётчик и переписывание ссылок.**
Счётчик `n` — **сквозной по всему документу**, начинается с 1.
В каждом блоке найти все `<i …class="…yfm-term_title…"…>ТЕКСТ</i>` по порядку.
Для каждого вхождения:
- ключ = значение `aria-controls`, иначе `term-key` + `_element`;
- если `defsMap[ключ]` отсутствует — **термин не трогаем** (нет `<sup>`, нет записи);
- иначе `n` = текущее значение, затем `n++`;
- термин превращается в ссылку вперёд (сам `<i>` сохраняется ради подчёркивания):
  ```html
  <i class="yfm yfm-term_title" …>ТЕКСТ</i><sup class="pdf-term-ref"><a href="#fn-{n}">{n}</a></sup>
  ```
- запомнить для текущей страницы пару `{n, ключ}`.

**Шаг 3. Вставка блока сносок в конец страницы.**
Если на странице есть записи — построить блок и вставить его **перед последним
`</div>`** блока-обёртки (это её закрывающий тег: внутренний контент сбалансирован,
поэтому последний `</div>` в чанке закрывает именно обёртку):

```html
<section class="pdf-footnotes">
  <ol>
    <li id="fn-{n}">{n}. CONTENT_из_defsMap[ключ]</li>
    …
  </ol>
</section>
```

Нумерация сквозная: на странице 2 сноски продолжают нумерацию страницы 1.
`id="fn-{n}"` глобально уникален (n сквозной), совпадает с `href="#fn-{n}"` термина.
Якорные `#`-ссылки Chromium сохраняет как внутренние переходы в PDF (тем же
механизмом уже работают TOC и pdf-ссылки).

### Извлечение атрибутов

Из открывающего тега `<i …>` атрибуты достаются под-регэкспом независимо от их
порядка: `aria-controls="([^"]*)"` и `term-key="([^"]*)"`.

### CSS (`PDF_STYLE_OVERRIDE` в `src/generate/constants.ts`)

Заменить ранее добавленный блок `.yfm-term_dfn` (хак `open`/block больше не нужен,
инлайновые `<dfn>` удаляются) на:

```css
/* прижать сноски к низу страницы, если контента мало */
.pdf .pdf-page-wrapper:has(.pdf-footnotes) {
    display: flex;
    flex-direction: column;
    min-height: 100vh;
}
.pdf .pdf-footnotes {
    margin-top: auto;          /* перебивается при коротком контенте — прижимает к низу */
    break-inside: avoid;
    page-break-inside: avoid;
    border-top: 1px solid rgba(0, 0, 0, 0.2);
    padding-top: 8px;
    font-size: 0.85em;
}
.pdf .pdf-footnotes ol { margin: 0; padding-left: 1.5em; }
.pdf .pdf-footnotes li { margin: 4px 0; }
.pdf .pdf-term-ref { font-size: 0.75em; }
.pdf .pdf-term-ref a { text-decoration: none; }

/* визуальный маркер термина */
.pdf .yfm .yfm-term_title {
    text-decoration: underline;
    text-decoration-style: dotted;
    text-decoration-thickness: 1px;
    text-underline-offset: 2px;
}
```

Новое flex-правило для `.pdf-page-wrapper` разместить **раньше** существующего
правила `:has(.yfm-page-constructor[data-hydrated="false"]) { display: none; }`,
чтобы `display: none` для скрытых обёрток выигрывал по порядку каскада.

## Краевые случаи и риски

1. `min-height: 100vh` в печати Chromium ≈ высота листа, но не пиксель-в-пиксель —
   возможна небольшая подстройка (фиксированная высота вместо `vh`). Главное к
   проверке глазами: короткие страницы (сноски должны уезжать вниз листа).
2. Длинная страница (>1 листа): блок сносок с `break-inside: avoid` через
   `margin-top: auto` уезжает к низу последнего листа раздела — желаемое поведение.
3. `<i>…</i>` матчится нежадно (термины не вкладывают `<i>`); `<dfn>…</dfn>` —
   нежадно до первого `</dfn>`.
4. Термин без определения в `defsMap` оставляем как есть (без `<sup>` и без записи),
   чтобы не плодить битые ссылки.
5. Термин в код-блоке без `aria-controls` — ключ берётся из `term-key` + `_element`.

## Тестирование

Юнит-тест `__tests__/terms.test.ts` (чистая строковая функция, запуск `npx jest terms`,
без тяжёлого integration-билда). Вход — html из двух `.pdf-page-wrapper` с терминами
и `<dfn>`. Проверки:

- исходные `<dfn class="yfm-term_dfn">` удалены из html;
- у каждого термина с определением появился `<sup class="pdf-term-ref"><a href="#fn-N">N</a></sup>`;
- в конце каждой страницы есть `<section class="pdf-footnotes">` с записями;
- нумерация сквозная (страница 2 продолжает нумерацию страницы 1);
- `id="fn-N"` записи совпадает с `href="#fn-N"` термина;
- повторное вхождение того же термина получает новый номер и новую запись;
- страница без терминов не получает блок сносок;
- термин без определения в `defsMap` не получает `<sup>`.

Дополнительно — ручная проверка PDF: переход по клику терм→сноска работает,
блок прижат к низу на короткой странице.
